• Guru Rubric

    What makes a guru?

    1. Confidence

    2. Youth

    3. Gender-neutral

    4. Pose

    5. Hair and eyebrows

    6. Glasses

    7. Piercings

    What can we variate?

    • Confidence
    • mostly positive expressions
    • always end in positive
    • Pose
    • arms*

    Arms are only moved when needed to hold props, i.e. the camera for photo guru.

    • Hair and eyebrows
    • color*
    • eyebrow angle

    Hair changes color on when dependent on a theme, usually a category - changing the color can change the overall feel which is something that should be maintained as soon as poeelble.

    • Glasses
    • shape*
    • color*

    See above.

    • Clothing
    • one or two piece outfit
    • hat
    • props*

    Clothing should remain neutral in both gender and theme unless specifically stated such as cowboy guru, etc.

    *dependent on theme

    How should colors be selected?

    • association with theme

    We chose a green color for transit to mimic traffic lights, red/orange for munchies which signifies a lot of different food sites, etc

    • color mix or palette
    • 25%-75% range
    • 30% tint for tiles/backgrounds

    Colors should adhere to the Uguru palette without straying too close to one color or another.

    • accents
    • derive from primary color
    • neutral palette

    Accent colors can be neutral colors such as  charcoal ,  slate ,  rocket ,  smoke , or white. They can also be tints/shades of the primary color, or even complementary colors derived from sites like Adobe Color.

    How should costumes and props be selected?

    Powerups are more of a personality variation  (costumes primary, props secondary)  while categories are more of an interest varation  (props primary, costumes secondary) .

    Costumes and props are more open-ended as long as they match with the overall theme selected.

    Props are generally rooted in reality. Props should follow a cloud motif if not reality-based because this allows more abstract props to be part of the theme.

    What does the composition look like?

    Sets of gurus should have the  same dimensions  and similar constrictions. For example, all the category loader gurus have the same width and height overall; all the category guru heads have the same height and width overall. The dimensions should be based off of the guru with the largest props/costumes. Gurus forms should be  arranged at center bottom  to remain grounded in reality and for optimal animation.

    Props can be layered in front or behind of the guru to create depth; they can be inside or outside of the constrictions inside the container.

    How should the gurus and powerups be animated?

    • default animation styles
    • soft and chill
    • opacity fades
    • bounce/jelly
    • draw lines

    Animation should start from our defaults and variate based on powerup themes. For example, GPA Guru uses a lot of fades and slides while Munchies Guru uses more jelly animations.

    Requirements:

    • States must at least include an onEnter
    • Other states that may be needed are onLoad and onExit

    1. Research

      • Do some research to get a better idea of the project at hand
        • →  List out what we like and how we can utilize it
      • Reasons
        • →  Makes the project more efficient by reducing the amount of backtracking
        • →  Gives us a better idea of best practices for animating
      • Good websites to research from
        • http://codepen.io
        • http://dribbble.com
        • http://awwwards.com
        • http://tympanus.net/codrops
        • http://onepagelove.com
      • Random Notes Regarding Animation Edge Cases
        • Applying edits to transition properties (via inline style) during a transition does not impact the animation. Of course, you can always set it null.
        • Same scenario but for animation properties, the changes DO apply. In fact the animation resets and restarts if you alter anything in the middle of the animation.

    2. Components

      • List out all the components required to complete the project
        • →  Specify if they require static work or updates
        • →  Discuss how to construct for optimal animation

    3. States

      • List out all the states required for this project
        • →  Specify animations per component listed in step 2
        • →  Loose timing to determine the order of component animation first per state

    4. Static

      • Set up base HTML and CSS to work from for the duration of the project
        • →  Make updates to the static and/or create new components in SVG/HTML format
          • →  Use whatever tools needed, but completing this step requires the updates to be pushed to the database
        • →  Gather components that are already usable
        • →  Create or update the HTML and SASS files with the new work and base directive implementation (i.e. sending the init state, on-enter, on-exit, on-click, etc)

    5. MVP Implementation

      • Start implementing base animations

    6. Detailed Implementation

      • Creating and implementing custom animations and new ones that don’t exist yet
        • →  All the animations we specified in step 3 unless it requires JS implementation or is too complex to complete in a reasonable amount of time, i.e. the slot machine flip for the splash tags

    7. Polish

      • Discuss the work from steps 6 and 7 and fine-tune animations based on feedback
        • →  Typically includes retiming and updates to states

    8. Bonus

      • Implement the custom animations that weren’t covered in step 6
        • →  If the animations turn out not to be feasible, provide a reasonable alternative that is easier to implement

  • Higher Level Directives Notes

    states > types

    • Internal:

      Default/custom states defined specific to the component, i.e. on-click

    • External:

      States defined by a component to react based on external components emitting messages

    • Synchronous:

      As one element's properties/animations/states are changing over time
    states > args
    • When a component responds to a state (modifies its own propertes in reaction to an internal/external/synchronous state msg)

      When a particular state activates, it can respond through these pre-defined means:

    • 1. class(es)
      [pre-defined]

      One to many classes injected into, each with configurable properties (see next section)

    • 2. animation(s)
      [pre-defined]

      One to many animations injected into, each with configurable properties (i.e time-state-func)

    • 3. custom CSS property tween(s)
      [via html attr]

      Clarifying 1-many properties to change over-time 
      tween:opacity|from:0,to:100,step:0.1
      or immediately 
      set:opacity:0.5

    • 4. Messaging
      [communicating]


      Communicating via our protocol to other elements [parents, children, siblings] via.
      • *the message you received (routing)
      • *the initiation/progress/completion of your reaction to the state received.
    states > defaults
    • - render [default, its pretty much a comment until you clarify to render]
    • - while-render [i.e loaders]
    • - on-init [after layout + paint is complete from opacity = 0, z-index:-1000]
    • - on-enter [how to enter the view port]
    • - on-exit [how to exit to view port]
    • - destroy [how to exit the DOM, back to comment]
    states > args > properties [unfinished]
    • Each state arg comes with modifieable properties

  • <head>

    <title>
    Defines the title of a document, shown in a browser's title bar or on the page's tab. Can only contain text.
    <base>
    Defines a default address or a default target for all links on a page. Specifies the base URL to use for relative URLs. There can only be one <base> element in a document.
    <link>
    Defines the relationship between a document and an external resource. Most often used for stylesheets.
    <meta>
    Defines metadata about an HTML document and other information that cannot be represented by one of the other <head> elements.
    <script>
    Defines a client-side script.
    <style>
    Defines style information for a document.
    <!--[if IE 6]>
    Can be used to format the page in Internet Explorer 5-9 only.

  • Rendering Notes

    Javascript  →  Style  →  Layout  →  Paint  →  Composite

    Javascript
    Handles work that results in visual changes.
    Style
    Applies CSS rules to appropriate elements based on matching selectors.
    Layout
    Calculates space and position of each element on the screen.
    Layout is skipped if a "paint-only" property like text color is used.
    Paint
    Draw calls for all elements, including text, colors, images, etc.
    Paint is skipped if the property doesn't require either layout or paint, such as transform.
    Composite
    Drawing elements on screen in correct order.
    Elements meant to be animated should be promoted with will-change: transform and/or transform: translateZ(0), which changes composite order.
    Rasterize
    Visually fills in pixels of all elements, including text, colors, images, etc.
    Rasterize is skipped if the property doesn't require either layout or paint, such as transform.

  • Animation Directive

    on-init

    Applies ONE initialize property, primarily for cases where you want to set the opacity, visibility, display immediately after the element renders 


    •  
       on-init="opacity:0" 
       
    •  
       on-init="opacity:0.5" 
       
    •  
       on-init="visibility:hidden" 
       
    •  
       on-init="display:none"
    class-on-click
    Adds class(es) when element is clicked
    activate-on-class
    Activates when class is added to element
    activate-on-class-delay
    Delays element being activated
    class-on-activate
    Adds class(es) when element is activated
    class-on-activate-delay
    Delays class(es) being added
    draw

    Draws any singular line, polgyon, path, rect, circle with these options:


    • 1. draw-duration: 1000ms, 1s (default)
    • 2. draw-start-frame: 0
    • 3. draw-delay: 1000ms, 1ms [default == 0]
    draw-shapes

    Draws all svg shapes within all childnodes (at any depth) of this element


    Applies the same options as draw, but can be declared on the same element 'draw-shape' is declared on

    If a child has "Draw" already, it's attributes will already be ignored, and the child's draw preferences (i.e. drawStartFrame), will override that of it's parent with the "draw-shape" directive

    draw-ignore


    Skips the request for a particular element, or a group of elements (just apply to highest parent where the children do not need to be drawn) a parent container that has decladed "draw-shapes"

  • UX Funnel

    Admin Flows
    Refers to admin-side groups that encompass large sections of the app. For example, the request flow starts at the student home page and ends at the public-facing request. Admin Flows and User Stories sometimes are the same, but with different focuses. Admin Flows tend to be more general than User Stories.
    User Stories
    Refers to an action a user might take in our app and the way the action is fulfilled. For example, “Guru checks ranking” is a user story.
    Scenes
    Refers to animated groups such as the projector. May be nested inside containers.
    Layouts
    Refers to the way that containers and components (including groups) are arranged on a given page. There are main layouts such as the ⅓ header and ⅓ sidebar as well as sublayouts like grid or tile layouts.
    Container Groups
    Refers to a group of containers that doesn’t take up an entire page but are larger and contain more detail than containers. An example of a container group would be the projector on the home page.
    Containers
    Refers to a group of components. Can also refer to layouts or large-format SVGs intended for animation.
    Component Groups
    Refers to multiple components grouped together that aren’t large enough or complex enough to be containers. For example, when you click “Logout” on the desktop navbar, there’s two buttons for confirming or canceling. Those two buttons together are a component group but not a container by itself. Other examples include the tile group on the dashboards, profile cards, or the request tile.
    Components
    Refers to elements made up of assets, text, and/or photos, such as buttons, progress bars, or user icons. Components are normally used as building blocks for component groups or containers.
    Assets
    Refers to photos and SVG icons used to make components. An asset cannot be used by itself in a container or component group.
